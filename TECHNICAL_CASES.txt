Case 001 - 2026-06-18 - Troll dawn never arrives after the troll encounter
---------------------------------------------------------------------------

Symptom
-------
After the player reaches the trolls and then leaves the clearing, repeated
"wait" commands in another location sometimes never trigger "Day dawns.".
The problem was especially visible in debug flows started with "jump trolls".

Observed behavior
-----------------
- The player could enter trolls_clearing and see the live trolls.
- The intro scene for the troll encounter sometimes did not fully initialize.
- The offscreen dawn timer never progressed.
- Temporary debug output showed:
  - visitedTrollsClearing stayed "no"
  - waitCounter stayed at 0 while outside the clearing
  - forceEncounter stayed "yes"
- A temporary debug catch exposed the browser error:
  "The quota has been exceeded."

Root cause
----------
There were two related problems.

1. The "jump trolls" checkpoint placed the player too deep into the scene flow.
   That made the encounter depend on follow-up state and timing that were not
   always initialized consistently.

2. The real blocker was autosave failure during the troll encounter setup.
   In checkTrollsClearing(), the code attempted:

   game.recordAutosave("before facing the trolls", { key: "hazard:trolls:room" });

   before the encounter state was fully committed.

   If localStorage was full, recordAutosave() threw a quota error. That
   interrupted the function before visitedTrollsClearing, waitCounter, and the
   other troll-scene flags were safely finalized. As a result, the game looked
   like the player had met the trolls, but the offscreen dawn logic still
   thought the encounter had never properly started.

Resolution
----------
The final fix had four parts.

1. "jump trolls" now lands on the troll approach ("dreary"), one move before
   the clearing, instead of placing the player directly inside trolls_clearing.

2. The jump also sets a temporary force flag:
   debugforcetrollencounter
   so the first entry into trolls_clearing always starts the intended scene.

3. Movement into trolls_clearing now decides up front whether the troll
   encounter must trigger, and directly runs checkTrollsClearing() on entry
   instead of relying only on later generic post-move checks.

4. checkTrollsClearing() now commits critical encounter state before attempting
   the autosave:
   - visitedTrollsClearing = true
   - debugforcetrollencounter = false
   - waitCounter = 0
   - troll warning state reset

   The autosave is then attempted afterward, so save failure cannot break the
   encounter state.

5. recordAutosave() was hardened against localStorage quota failures.
   If the browser reports quota exhaustion, the code now:
   - recognizes the quota error
   - removes older autosaves
   - retries saving
   - returns false safely if it still cannot save
   - does not let the exception break game flow

Files touched
-------------
- game.js

Key implementation notes
------------------------
- The robust fix was not "reset the flag less often". The issue was that the
  flag was never being committed because the autosave exception interrupted the
  encounter setup.
- For hazardous-scene triggers, gameplay state should be committed before any
  best-effort persistence layer like localStorage autosave.
- When debugging browser-only state bugs, temporary in-output debug lines were
  useful:
  - troll-debug
  - troll-debug-entry
  - debug-error
  They were removed after confirmation.

Future debugging advice
-----------------------
- If a scene seems to start visually but downstream timers do not move, check
  whether a browser-side persistence call is throwing mid-scene.
- If a user reports behavior that cannot be reproduced locally, add temporary
  in-output debug lines rather than relying only on the JS console.
- For other hazard scenes, review whether autosave happens before or after the
  scene state is committed.


Case 002 - 2026-06-18 - Clicking a world-map location would not open its local map
------------------------------------------------------------------------------------

Symptom
-------
Using the mouse to click a clickable location on the scene map, such as Misty
Mountains, no longer opened that location's dedicated local map. Directional
movement still preserved the intended map behavior while travelling.

Observed behavior
-----------------
- The click handler still received the map hotspot click.
- The target region id (for example `mountains`) was detected correctly.
- After the click, the map immediately stayed on or snapped back to `world`
  instead of opening the requested local scope.
- Movement-based map updates still worked, because auto-follow based on current
  room remained active.

Root cause
----------
The click path and the movement path were sharing a single scope-reconciliation
flow.

- `openSceneMapScope(regionId)` set the scope after a click, but the later map
  render reconciliation still treated the current room as authoritative.
- That reconciliation was correct for auto-follow mode, but wrong for manual
  click-open behavior.
- As a result, clicking a location did not persist a manual scope choice.

Resolution
----------
The fix introduced an explicit manual scope mode for the scene map.

1. Added `sceneMapManualScope` to distinguish:
   - movement-driven auto-follow map behavior
   - click-opened manual map behavior

2. When the player clicks a map hotspot:
   - the exact clicked region becomes the scope
   - `sceneMapAutoFollow = false`
   - `sceneMapManualScope = true`

3. During map render:
   - if `sceneMapManualScope` is true, the chosen scope is preserved
   - otherwise the existing room-based reconciliation logic still runs

4. `showMap()` and other movement-driven map openings explicitly reset manual
   mode, preserving the existing behavior when navigating by directions.

Files touched
-------------
- game.js

Future debugging advice
-----------------------
- If clicks appear to work visually but do not change the map state, inspect
  scope reconciliation after the click, not only the click handler itself.
- Keep click-driven map mode and movement-driven auto-follow mode distinct.

Case 004 - 2026-06-18 - Local maps missed immediate parent/world connections shown by the editor
--------------------------------------------------------------------------------------------------

Symptom
-------
Inside a local map such as Rivendell, the game only showed anonymous dashed
exit stubs for out-of-scope links. The immediate parent/world connections that
were visible in the map editor, such as Hidden Valley Path and Misty Mountains,
were not rendered in the in-game map.

Root cause
----------
The in-game scene-map model built only `exitStubs` for connections leaving the
current scope. Unlike the editor, it never promoted those links into explicit
external exits resolved against the parent scope, so the renderer had no
destination labels to show.

Resolution
----------
The fix aligned the game renderer with the editor logic.

1. Added a scope-aware destination resolver for rooms outside the current
   local map, so the game can map a connection to the correct parent/world
   node label.

Case 005 - 2026-06-18 - Golden-path autoplay test hid seed-sensitive Gollum and Smaug failures
-----------------------------------------------------------------------------------------------

Symptom
-------
The existing autoplay regression in `scripts/parser-tests.js` passed reliably,
but separate seeded batch runs could still fail in "real" autoplay playthroughs.
Observed seeded failures clustered in two places:

- Gollum at `deep_dark_lake`
- Smaug / Bard handoff in Erebor

Root cause
----------
The existing regression only exercised one deterministic golden path:

- it forced `Math.random = () => 0.99`
- it ran a single autoplay route
- it did not vary `storySeed`

That meant the suite proved one winning path existed, but it did not probe the
seed-sensitive branches that affect:

- Gollum riddle selection and encounter flow
- Erebor/Smaug timing and state progression

Resolution
----------
Added a seeded autoplay batch mode to `scripts/parser-tests.js` so autoplay can
be probed across many reproducible runs without changing the default suite.

Usage
-----
- `node scripts/parser-tests.js --autoplay-batch-only --autoplay-batch=100`
- optional flags:
  - `--autoplay-seed-start=<n>`
  - `--autoplay-step-limit=<n>`
  - `--autoplay-batch-strict`

The batch runner now:

- assigns a reproducible PRNG per seed
- resets the game and reapplies the same seed to `storySeed`
- recreates seed-sensitive encounter state such as Gollum
- executes autoplay up to a configurable step limit
- prints a per-seed report with outcome classification and last visible line

Observed report after adding the harness
---------------------------------------
For seeds `1-100` with step limit `500`, the batch produced:

- 76 victories
- 13 `stuck_gollum`
- 10 `death_smaug`
- 1 generic `step_limit`

Representative failing seeds
----------------------------
- Gollum stalls: `7, 11, 20, 31, 32, 33, 45, 69, 74, 76, 80, 87, 94`
- Smaug deaths: `21, 23, 37, 41, 62, 63, 77, 79, 83, 90`
- Other step-limit: `27`

Files touched
-------------
- `scripts/parser-tests.js`

Future debugging advice
-----------------------
- Treat the golden-path autoplay test as a smoke test, not as proof of
  autoplay robustness.
- When changing autoplay logic for Gollum or Erebor, rerun the seeded batch and
  promote a few representative failing seeds into targeted regressions.
- The last output line in the batch report is useful for spotting whether a run
  is failing because autoplay is speaking to no one, missing a state
  transition, or looping after a room-state change.

2. Local map models now build `externalExits` for immediate off-scope
   connections and keep their destination labels.

3. The SVG renderer now draws those exits as labeled pills with direction
   badges, matching the editor behavior more closely.

4. Anonymous exit stubs are filtered out when a labeled external exit already
   represents the same source node and direction, avoiding duplicate marks.

Files touched
-------------
- game.js

Future debugging advice
-----------------------
- If the editor shows a cross-scope map link that the game does not, compare
  the model-building step first, not only the SVG rendering step.
- Keep unexplored internal exits (`exitStubs`) separate from parent-scope
  exits with known destinations, because they serve different UI purposes.


Case 005 - 2026-06-18 - Map mode needed a moving current-cell frame during travel
----------------------------------------------------------------------------------

Symptom
-------
With the explored map open, changing location immediately snapped the current
highlight to the new cell. There was no sense of the selection frame travelling
from the old room to the new one.

Root cause
----------
The current-room frame was baked directly into the generated SVG. Because the
map is rendered as a data-URL image, that static highlight could only jump when
the SVG was rebuilt for the new room.

Resolution
----------
The fix moved the current-room frame out of the SVG and into a DOM overlay.

1. Added a `.scene-map-current-indicator` element inside the map stage.
2. Stored a pending map-travel animation whenever the player moves while the
   scene map is visible.
3. Reused `editorMeta.nodeCenters` to resolve both the old and new node boxes
   in the freshly rendered scope.
4. Animated the overlay frame from the previous node to the new node when both
   rooms are visible in the same rendered map; otherwise it safely falls back
   to an immediate placement.
5. Cleared animation timers/state whenever the map closes, changes scope, or
   other UI flows reset the map.

Files touched
-------------
- game.js
- style.css
- scripts/parser-tests.js

Future debugging advice
-----------------------
- If a map movement animation fails only on some routes, check whether both the
  source room and destination room resolve to visible nodes in the same scope.
- For map UI effects layered over the SVG, prefer DOM overlays fed by
  `editorMeta` coordinates instead of re-encoding the SVG on every frame.


Case 006 - 2026-06-18 - Entry into a local-map region skipped the travel animation
-----------------------------------------------------------------------------------

Symptom
-------
When the player moved into a location that owns its own local map, the scene
map immediately drilled down into that local map. The current-location frame
did not have time to animate across the shared parent map first.

Root cause
----------
Auto-follow resolved the destination scope immediately from the new room.
That meant moves such as world -> Beorn's House rebuilt the map straight into
`beorn`, so the source node only existed on the old world map and the target
only existed on the new local map. There was no common rendered scope for the
frame animation.

Resolution
----------
The fix introduced a two-step travel plan for map-follow transitions.

1. On movement with the map open, the game now stores:
   - `holdScope`: the shared scope where both endpoints can be represented
   - `finalScope`: the destination scope after the animation
   - whether a scope swap is pending after the animation

2. While the animation is active, scene-map rendering prefers `holdScope`
   instead of immediately auto-following into the destination local map.

3. When the frame animation completes, the map automatically re-renders into
   `finalScope`, so the drilldown happens only after the motion is visible.

4. Parser-test DOM stubs were updated with a minimal `querySelector()` for the
   current-frame overlay, allowing the delayed drilldown behavior to be tested.

Files touched
-------------
- game.js
- scripts/parser-tests.js

Future debugging advice
-----------------------
- If a move crosses map boundaries, compute an intermediate shared scope first;
  otherwise animated markers will appear to teleport.
- In DOM-light test harnesses, map overlays that are inserted via `innerHTML`
  may need lightweight query helpers or the animation path will not execute.


Case 007 - 2026-06-18 - Keyboard travel kept force-centering the map viewport
------------------------------------------------------------------------------

Symptom
-------
With the map open, moving by commands such as `n`, `e`, and similar always
recentered the scroll viewport on the new location. This made the map feel
jumpy even when the destination was already visible.

Root cause
----------
After each movement render, the scene map fell back to `centerSceneMapOnRoom()`
whenever no explicit viewport anchor was present. That was correct for first
opening the map, but wrong for movement updates where the existing scroll
context should be preserved.

There was also a second render during animated travel. Without preserving the
viewport between those renders, a minimal visibility adjustment could still be
overwritten by a later full recenter.

Resolution
----------
The fix introduced a movement-specific viewport mode.

1. Command travel now marks the next map render as `travel` mode.
2. In `travel` mode, the map checks whether the destination node is already
   inside the current viewport.
3. If the destination is visible, the scroll position is left unchanged.
4. If it is outside the viewport, the map scrolls only enough to bring it back
   into view with a small margin, instead of fully centering it.
5. While a travel animation is active, the current viewport is captured and
   restored across follow-up renders so the gentle adjustment is not replaced
   by a second full recenter.

Files touched
-------------
- game.js
- scripts/parser-tests.js

Future debugging advice
-----------------------
- Keep “initial open” viewport behavior separate from “movement follow” viewport
  behavior; they need different heuristics.
- If a scroll tweak seems to work and then disappears, look for a second render
  in the same interaction rather than only the first viewport calculation.


Case 008 - 2026-06-18 - Restored map viewport could still leave the current room offscreen
-------------------------------------------------------------------------------------------

Symptom
-------
In some already-rendered map states, the current location could still end up
outside the visible viewport. The map image was present, but the player had to
drag the scrollbars manually before the active location became visible.

Root cause
----------
Viewport restoration and centering were treated as the final layout step.
If a stale viewport anchor restored the scroll position to an area that no
longer contained the current room, the render completed without any final
sanity check that the tracked location was actually visible on screen.

Resolution
----------
Added a final visibility safety pass after map positioning.

1. After restoring or choosing the viewport, auto-follow renders now run
   `ensureSceneMapRoomVisible()` as a final guard.
2. If the current room is already visible, nothing moves.
3. If it is offscreen, the map nudges the viewport just enough to bring it
   back into view.
4. Added a regression test that forces a stale viewport anchor and confirms
   the map rescues the current location without manual scrolling.

Files touched
-------------
- game.js
- scripts/parser-tests.js

Future debugging advice
-----------------------
- For scrollable map UIs, restoration and centering are not enough on their
  own; add a final “is the tracked target actually visible?” check.
- If users report “the map is there but the place I am in is missing”, inspect
  stale viewport anchors before inspecting the SVG or node placement logic.


Case 010 - 2026-06-18 - Scope swaps needed to preserve the full current-location frame, not only the node center
---------------------------------------------------------------------------------------------------------------

Symptom
-------
When the player entered a location with its own local map, or backed out from a
child map to its parent, the new scope could reopen with the active location
feeling re-centered or slightly misaligned. The highlighted frame did not keep
the same visual continuity across the scope change.

Root cause
----------
Viewport rescue logic treated the current location mostly as a point target.
That was good enough to keep the node center visible, but not enough to keep
the full current-location frame safely inside the viewport with the expected
visual continuity during scope transitions.

Resolution
----------
The map viewport now treats the active location as its full indicator box.

1. Added a scaled indicator-box helper for the current location.
2. Visibility checks and gentle follow nudges now keep the whole frame inside
   the viewport margins, not just the center point.
3. Scope-transition regression tests were tightened so drilldown/back behavior
   verifies:
   - the captured anchor is restored before any extra pan
   - any follow-up pan only happens as a minimal rescue
   - the full indicator box ends up on screen after settling
4. The “steady scroll” regression test was updated to use a viewport setup
   where the full destination frame is already visible, matching the new rule.

Files touched
-------------
- game.js
- scripts/parser-tests.js

Future debugging advice
-----------------------
- For map transitions between parent and child scopes, validate frame-box
  visibility rather than only node-center visibility.
- If a scope swap appears to open “too centered”, inspect whether the viewport
  rescue step is reasoning about the box edges or only the point center.


Case 009 - 2026-06-18 - Cross-scope opening should not force a pan if the target is already visible
-----------------------------------------------------------------------------------------------------

Symptom
-------
After adding cross-scope opening from the source-node position, the new scope
could still perform a scripted pan toward its centered viewport even when the
current location was already visible as opened.

Root cause
----------
The first cross-scope implementation always panned toward the scope's centered
target after opening from the source-node anchor. That made the transition more
active than the normal map behavior, which only moves when the current room
would otherwise be offscreen.

Resolution
----------
The scope-transition pan now reuses the same visibility rule as normal travel.

1. The new scope still opens from the source-node position.
2. The follow-up pan no longer targets full centering by default.
3. Instead, it computes the same minimal visibility-adjusted viewport used by
   standard map travel.
4. If the current room is already visible, no extra pan runs at all.

- scripts/parser-tests.js

Future debugging advice
-----------------------
- Keep special transitions visually consistent with the base navigation rule;
  otherwise map behavior will feel unpredictable between similar moves.


Case 011 - 2026-06-18 - Small local maps could not preserve scope-swap anchors
--------------------------------------------------------------------------------

Symptom
-------
Moving from a world-map node into a small local map, such as Narrow Dangerous
Path -> Beorn's House, still appeared to snap instantly into the local map's
default position instead of reopening from the same on-screen frame position.

Root cause
----------
There were two linked blockers.

1. Some local maps at the active zoom were smaller than the visible viewport.
   Example: Beorn's House at 60% rendered to about 660x491 inside an ~800x500
   viewport. That left no horizontal scroll range, so preserving the incoming
   world-map offset was impossible unless the map was given extra layout space.

2. During scope swap, a stale viewport anchor from the previous world scope
   could still be present. Because `prepareSceneMapScopeTransitionAnchor()`
   aborted when *any* anchor existed, the new child-scope anchor was never
   installed, so the render fell back to the clamped local-map position.

Resolution
----------
1. Scene-map layout now adds dynamic stage padding when needed, so small maps
   can still preserve the desired focus offset inside the viewport.
2. The stage offset is included in:
   - viewport capture/restore
   - point-to-viewport offset calculations
   - scaled indicator-box calculations
3. Scope-transition anchor preparation now ignores stale anchors from other
   scopes and replaces them with the correct anchor for the target scope.

Files touched
-------------
- game.js
- style.css
- scripts/parser-tests.js

Future debugging advice
-----------------------
- If a scope transition seems to ignore its anchor, check both:
  - whether the target scope has any actual scrollable range
  - whether an old anchor from another scope is blocking replacement


Case 012 - 2026-06-18 - Local in-game maps could collapse around visited rooms instead of keeping editor geometry
--------------------------------------------------------------------------------------------------------------------

Symptom
-------
Some local maps in the game, including Misty Mountains, did not match the map
editor. The editor showed the correct full local layout footprint, but the
in-game map collapsed around only the visited rooms. That changed the apparent
shape and proportions of the local map compared with the editor reference.

Root cause
----------
The in-game local-map builder used `visitedRooms` both to decide which local
rooms to render and to derive the effective local-map footprint. The editor,
instead, always used the full scope layout from `assets/map-layout-data.js` as
the positioning reference.

There was also a child-portal mismatch:
- the editor always attached inline child-region portals to the host node
- the game only added those portals after at least one child room was visited

Resolution
----------
Aligned the game local-map model with the editor behavior while preserving
exploration fog.

1. Local maps still render only visited rooms and their reachable metadata.
2. The renderer now derives local-map bounds from the full scope layout, so
   the visible subset sits in the same geometry and scale used by the editor.
3. Inline child-region portals now appear whenever the parent scope defines
   them, matching the editor.
4. Added a regression test for Misty Mountains that verifies:
   - unvisited rooms stay hidden
   - the local map still keeps the full editor-sized footprint

Files touched
-------------
- game.js
- scripts/parser-tests.js

Future debugging advice
-----------------------
- When the editor and game disagree on a local map, first compare the room set
  that each model builder includes before debugging connector geometry.
- Treat the editor layout as the source of truth for local scope membership and
  nested portals unless exploration-based pruning is an explicit design choice.


Case 013 - 2026-06-18 - Local-map connector waypoints used editor coordinates without game-space remapping
------------------------------------------------------------------------------------------------------------

Symptom
-------
After restoring the correct full local-map footprint, node boxes matched the
editor much better, but some local-map connectors still had the wrong final
shape in the game. Their bends and jogs did not line up with the editor.

Root cause
----------
The decisive mismatch was in lane-id resolution for one-way non-compass links.

- The editor stores a lane like:
  `ow:room:deep_misty_valley_2>room:narrow_path_6:up:`
- The game generated:
  `ow:room:deep_misty_valley_2>room:narrow_path_6:up:down`

Because the ids did not match, the game failed to load the lane-specific
override from `assets/map-layout-data.js` and silently fell back to auto
routing for that connector.

Box/anchor metric differences were a secondary mismatch, but they were not the
root cause of the persistent Deep Misty Valley connector error.

Resolution
----------
1. Added editor-compatible fallback matching for one-way lane ids such as
   `up:`, `down:`, `inside:`, `outside:`, `forward:`, and `back:`.
2. When a direct lane-id lookup misses, the game now also checks the legacy
   editor form with a blank target direction for those one-way lanes.
3. This allows the renderer to load the correct lane-specific waypoint and
   anchor overrides instead of falling back to auto routing.

Files touched
-------------
- game.js
- scripts/parser-tests.js

Future debugging advice
-----------------------
- If node positions look right but connector bends do not, inspect the
  lane-id matching first, especially for one-way `up/down`-style connectors.
- Saved layout overrides can only be shared between tools if both sides agree
  on lane-id normalization as well as SVG coordinate transforms.

Case 015 - 2026-06-18 - Scene-map regression tests still assumed world-first opening and complete-map drilldown lost full reveal
-------------------------------------------------------------------------------------------------------------------------------

Symptom
-------
After removing the obsolete inline map-editor tests, `node scripts/parser-tests.js`
still failed on several scene-map cases.

The visible mismatches were:
- some tests still expected `map` to open the world overview first
- some tests expected immediate re-drilldown into a local scope while the
  current-room frame animation was still in progress
- `complete map` could open the full world overview, but manually opening a
  local scope from there silently dropped the full-reveal mode

Root cause
----------
There were two different issues mixed together.

1. The scene-map behavior had evolved, but older tests still described a
   previous model:
   - `showMap()` now opens the current room's deepest local scope first
   - `sceneMapBack()` climbs parent scopes one level at a time
   - boundary-crossing auto-follow can intentionally hold the parent/world
     scope briefly before drilling back down after the frame animation

2. `openSceneMapScope()` always forced:

   `sceneMapShowAll = false`

   for non-world scopes. That was fine for ordinary explored-map browsing, but
   wrong when the player started from `complete map`. It meant a manual
   drilldown into a full local map immediately reverted to visited-only rules.

Resolution
----------
The fix had two parts.

1. Preserved `sceneMapShowAll` when manually opening a child scope.
   That lets `complete map` keep its full-reveal behavior while drilling into
   local maps such as Long Lake.

2. Updated the parser tests to match the current scene-map contract:
   - `map` can open directly to a local scope such as Bilbo's Home or Erebor
   - `back` climbs the scope hierarchy before closing the overlay
   - some re-drilldowns are intentionally delayed until movement animation
     settles
   - world/local drilldown tests now use valid jump checkpoints and setups
     that reflect the current map progression

Files touched
-------------
- game.js
- scripts/parser-tests.js

Verification
------------
- `node scripts/parser-tests.js`
- Result: `1100 parser/dialogue tests passed.`

Future debugging advice
-----------------------
- When scene-map tests fail, check first whether the scope semantics changed:

Case 091 - 2026-06-21 - Consecutive narrated lines could stall after the first spoken sentence
-----------------------------------------------------------------------------------------------

Symptom
-------

----------

- the remaining queue never resumed

----------
- Kept consecutive queued narration on the same generation unless there is a
  real interrupt/cancel event
- Preserved generation invalidation only for genuine interruption flows
- Added regression coverage for the exact case of multiple consecutive narrated
  lines arriving while the first line is still speaking

-------
In the autoplay lab solution spine, some real fatal branches were still absent
even after the engine and the lab agreed on the underlying death trigger.

The clearest example was the Gollum pocket-question route:

- the winning path reached `say to gollum "what have i got in my pocket"`
- waiting immediately afterward was genuinely fatal in `game.js`
- but the spine only displayed branches for earlier riddle answers, not the
  post-step death branch after the pocket question

Root cause
----------
`collectFatalBranchesFromSpine()` only explored branch candidates from the
state *before* each winning command.

That meant the spine could show:
- "instead of this command, do X"

but it could not show:
- "after this command, doing X is fatal"

So any death branch whose trigger lived in the post-step state was invisible
to the user.

Resolution
----------
Extended the solution-spine collector and renderer to support both branch
timings:

- `before`: fatal branch instead of the canonical winning step
- `after`: fatal branch immediately after the canonical winning step

The UI now labels them explicitly as `Instead of ...` or `After ...`, and the
regression suite includes a source guardrail that checks this dual-mode branch
support stays present.

Files touched
-------------
- `autoplay-lab.js`
- `scripts/parser-tests.js`
- `TECHNICAL_CASES.txt`

Future debugging advice
-----------------------
- For audit tools built around a canonical path, always test divergence from
  both sides of each step:
  - pre-step alternatives
  - post-step consequences
- If a player reports "the death exists but the spine does not show it",
  inspect whether the fatal trigger lives in the state after the winning
  command rather than before it.


Case 144 - 2026-06-26 - Solution spine replays drifted from the winning run because the lab seeded Math.random but not storySeed
-------------------------------------------------------------------------------------------------------------------------------

Symptom
-------
The solution spine could still miss obviously real death branches, even after
support was added for `After ...` divergences.

In the Gollum route this showed up as:

- the winning path in the lab displayed one pair of correct riddle answers
- the branch replay rebuilt a different riddle sequence
- the spine then surfaced odd Gollum branches and still missed the fatal
  `After "what have i got in my pocket": wait` branch

Root cause
----------
The lab wrapped simulations in a seeded `Math.random`, but `resetToPreset()`
still let the game create a fresh internal `storySeed` during `restartGame()`.

Many story-critical choices in `game.js` are keyed off `storySeed`, including:

- Gollum riddle selection
- narrative variants
- combat/death text choices

So two lab replays could share the same RNG seed while still landing in
different narrative states.

Resolution
----------
Updated `resetToPreset()` so the lab explicitly applies the current run seed as
`game.storySeed` immediately after `restartGame()` and before the preset logic
runs.

All seeded lab replays now rebuild the same story-state family as the original
run, and the regression suite includes a guardrail checking that the reset path
continues to preserve story-seed alignment.

Files touched
-------------
- `autoplay-lab.js`
- `scripts/parser-tests.js`
- `TECHNICAL_CASES.txt`

Future debugging advice
-----------------------
- In this codebase, "same seed" means more than patching `Math.random`; any
  replay or audit tool must also preserve `game.storySeed`.
- If a seeded replay produces different riddles, different ambience, or
  different death text from the source run, inspect `storySeed` before looking
  at higher-level branch logic.
- Also check for seed-derived substate already materialized during
  initialization. In the Gollum chapter, `gollumState.riddleIds` had to be
  regenerated after reseeding, not merely left in place.


Case 145 - 2026-06-26 - Late tunnel deaths were mislabeled as Gollum because solution-spine classification trusted stale tail text
-------------------------------------------------------------------------------------------------------------------------------

Symptom
-------
In the solution spine, some deaths later in the goblin tunnels were still
labeled `Death by Gollum` even though the visible fatal scene was clearly a
goblin combat death.

Typical example:

- the tail still contained a recent Gollum pursuit echo
- the actual death line described Bilbo being killed by a disgusting goblin
- the spine still showed `Death by Gollum`

Root cause
----------
The lab's death classifier evaluated `death_gollum` before the goblin death
targets and allowed it to match any fatal branch whose recent output still
contained the word `Gollum`.

That was too broad for long tunnel runs, because Gollum pursuit echoes can
remain inside the last few output lines even when the true death cause is now a
different encounter.

Resolution
----------
Added `deathImage` to the scenario context and tightened the death matchers:

- Gollum deaths now prefer Gollum-specific death images or explicit Gollum
  fatal text
- hulking-goblin deaths can match their dedicated death image
- generic goblin deaths can match goblin-specific death images

This made the label follow the actual fatal scene instead of a stale mention in
the recent output tail.

Files touched
-------------
- `autoplay-lab.js`
- `scripts/parser-tests.js`
- `TECHNICAL_CASES.txt`

Future debugging advice
-----------------------
- For any audit UI that labels causes of death, prefer structured signals such
  as death-image ids or dedicated fatal markers over broad keyword matches in
  recent transcript text.
- If a late-branch death is mislabeled, inspect whether the output tail is
  carrying narrative residue from an earlier chapter or pursuer.
